The Living NFT — Sprouting Visual Spec

Partially Superseded · 2026-04-20

The visual mechanic in this doc (6 stages, ERC-4906, stageURIs) still holds. The economic mechanic does not: sprouting no longer decays price — it raises the minimum ask floor (Fresh +0%, Leafing +10%, Sprouted +25%, Bloomed +50%, Harvest +100%, Eternal +200%). The "decayed price" language and any references to the asking price falling with sprouting are stale. See hot-potato-v4-mechanic-spec.html for the locked parameters.

1. The Short Answer

Yes. The Hot Potato NFT (Token ID 0) visually transforms as it sprouts. Each of the five stages has its own pre-generated image and its own set of traits. The transformation is visible everywhere — on your site, on OpenSea, in wallets, in the Base block explorer, in social-media link previews.

This isn't a cosmetic site-only trick. It's a genuinely dynamic NFT, and the standard that makes it work is ERC-4906 (the MetadataUpdate event). When the sprouting stage changes on-chain, the contract swaps the tokenURI and emits a MetadataUpdate(0) event. OpenSea watches for that event and re-fetches within minutes; most wallets honor it too.

Why this matters for the game: a sprouted potato on OpenSea is a public invitation. Anyone browsing the Base NFT ecosystem sees that the Hot Potato has sprouts, checks the listing, sees the decayed price, and clicks buy. The NFT art doing the marketing for you.

2. The Five Visual States

Five stages, five pre-rendered images, one final boss image. Each is pinned to IPFS at launch — the art itself is immutable; only the pointer changes.

🥔

STAGE 1 · FRESH

Day 0 – 3 · Post-sale baseline

The classic Hot Potato. Steaming, golden-brown, looking freshly-taken-from-the-oven. The image you see in every marketing asset.

Prompt direction: steaming russet potato, crispy skin, small blistered surface, visible heat shimmer, warm rim light. This is the hero shot.

🌱

STAGE 2 · LEAFING

Day 3 – 7 · First growth signal

A single tiny leaf has pushed through the skin. Steam is still rising but is thinner. The potato has been sitting a while.

Prompt direction: same potato, same angle, same lighting — with one small green pale-yellow shoot emerging from one eye. Minimal change; the player's eye notices it because it's the same composition, with one detail different.

🌿

STAGE 3 · SPROUTED

Day 7 – 14 · Price decay begins · +1 Sprout Boost active

Multiple shoots, the potato looks noticeably less hot. The skin has started to wrinkle. Steam is gone. A few small green leaves have unfurled.

Prompt direction: dimmer lighting, cooler color balance, visible dried skin, 3–5 green shoots at varying lengths, one small leaf just starting to unfold. Still recognizably the same potato.

🌸

STAGE 4 · BLOOMED

Day 14 – 21 · Accelerated decay · +2 Sprout Boost active

The potato is now more plant than food. Full leafy growth. Small purple-white potato flowers (real potatoes do flower) bloom at the tips. This is the dramatic image.

Prompt direction: overgrown potato half-obscured by foliage, several purple/white flowers, subtly lit as if on a windowsill. Emotional beat: "this thing has been forgotten for two weeks."

🌾

STAGE 5 · HARVEST READY

Day 21 – 30 · Holder can call harvest() · potato is practically a plant

The potato is a small tangled bush. New baby potatoes are beginning to form where roots have sprawled. The composition is almost inverted — the original potato is barely visible at the base of its own plant.

Prompt direction: potato plant in full bloom, parent potato visible but wrinkled and spent, soft halo of light, reverent mood. This image tells the holder "you grew this."

👑

STAGE 6 · ETERNAL (FINAL SEAL ONLY)

The Last Supper ending · 1/1 forever

Only ever generated once, if and when the game seals. A unique, never-reused image — the potato as final artifact. Could be a fully grown plant with the original potato as a gnarled root, lit like a reliquary.

Prompt direction: reliquary aesthetic, single perfect potato plant, flowering, dark velvet background, gold accents. Painted once. Pinned once. Never regenerated.

4. The Technical Approach

4.1 Three Options — And Why I'm Recommending Option B

ApproachProsCons
A. Static NFT, site-only animation Zero complexity OpenSea, wallets, Twitter previews all show the static image. Sprouting is invisible outside your site. Wastes the mechanic.
B. ERC-4906 + pre-generated IPFS images ← recommended Standard-compliant. OpenSea auto-refreshes. Images are immutable and pre-approved. Low gas. Clean, legible for collectors. Requires someone to trigger pokeStage() to update. Solved with a keeper bounty.
C. Fully dynamic tokenURI() returning computed metadata Always up-to-date without any trigger. Truly live. Requires base64-encoded SVG or on-chain JSON. Complex. Image has to be SVG or pulled from a dedicated metadata server. Not fit for AI-generated hero art.

4.2 Implementation Sketch

// ERC-4906 event (standardized metadata refresh signal)
event MetadataUpdate(uint256 _tokenId);

// Pre-uploaded IPFS URIs, one per stage
string[6] public stageURIs = [
    "ipfs://.../fresh.json",
    "ipfs://.../leafing.json",
    "ipfs://.../sprouted.json",
    "ipfs://.../bloomed.json",
    "ipfs://.../harvest.json",
    "ipfs://.../eternal.json"     // final seal only
];

uint8 public currentStage;    // 0-5

function tokenURI(uint256 tokenId) public view override returns (string memory) {
    if (tokenId == 0) return stageURIs[currentStage];
    return _souvenirURIs[tokenId];  // souvenirs have their own URIs
}

function pokeStage() external {
    uint8 newStage = _computeStage();    // derives from block.timestamp - lastSaleTime
    require(newStage != currentStage, "no change");
    currentStage = newStage;
    emit MetadataUpdate(0);              // OpenSea will refresh

    // Small bounty for the poker (covers their gas + tip)
    if (treasury >= 0.0005 ether) {
        treasury -= 0.0005 ether;
        payable(msg.sender).transfer(0.0005 ether);
    }
}

function buyPotato(uint256 newAsk) external payable {
    // ... normal buy logic ...
    lastSaleTime = block.timestamp;
    if (currentStage != 0) {
        currentStage = 0;
        emit MetadataUpdate(0);           // resets to Fresh on every sale
    }
}

4.3 Why Keepers Aren't A Weakness

The concern: "what if no one calls pokeStage() and the NFT stays stuck on the wrong image?" In practice this is self-resolving:

  • A small keeper bounty (0.0005 ETH) creates incentive for anyone to call it.
  • Your own backend or a simple Chainlink Automation job calls it every few hours as a free safety net.
  • Every buyPotato() call also runs the stage reset anyway, so the "worst case" is a short delay until the next buy or next pokeStage call.
  • And honestly — if an outdated NFT image goes viral on X because the stage hasn't updated in a few hours, that's free marketing, not a bug.

5. Souvenir Cross-Pollination (Bonus Idea)

Since you're generating souvenirs via FLUX + LoRA at every sale, you have the option to bake the parent potato's stage into the souvenir art. A souvenir minted while the potato was Bloomed could carry leafy accents; a souvenir minted during Harvest could be shown clutching a tiny flower.

This creates a new, collectable axis that rewards buying at unusual moments:

Souvenir Minted DuringVisual SignatureCollector Value

Add a trait parentPotatoStage to every souvenir's metadata. Collectors will hunt for "Bloomed-era" souvenirs the same way they hunt for early token IDs. Costs nothing extra to implement — you're already generating images at mint time, just vary the prompt based on currentStage.

7. Art Production Checklist

  • Generate 6 hero images (Stages 1–5 + the 1/1 Eternal) with your FLUX+LoRA pipeline. Aim for consistent composition, angle, and lighting so the progression feels unified rather than 6 unrelated images.
  • Pin all 6 to IPFS via Pinata at launch. Immutable.
  • Write 6 metadata JSON files (one per stage) with updated trait values. Pin those too.
  • Hardcode the 6 URIs into the contract as string[6] public stageURIs.
  • Implement pokeStage() with the keeper bounty.
  • Add a trait-update pass so the Current Holder, Days Since Sale, Asking Price, Pot Size traits update in the live metadata (this requires either a lightweight metadata server you host, or accept that these traits only update at stage boundaries).
  • Add parentPotatoStage to souvenir metadata — record on mint, expose as a trait.

Pre-launch gotcha to avoid: test the metadata refresh flow on Base Sepolia before mainnet. Trigger a fake stage change, watch OpenSea testnet refresh. If it doesn't pick up the new image within 10 minutes, your metadata JSON probably has a cache-busting issue (set correct image fields, avoid ?v= suffixes, make sure IPFS URIs are immutable and not pointing to mutable gateways).
